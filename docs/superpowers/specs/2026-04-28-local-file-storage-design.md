# Local File Storage Design

## Goal

Store uploaded appointment files and videos permanently on the VPS disk, then send email notifications with download links instead of SMTP attachments.

External storage services are out of scope. Files must not be deleted automatically.

## Current Behavior

`POST /api/send-email` accepts JSON and `multipart/form-data`.

When files are present, `multer.memoryStorage()` keeps them in memory and `nodemailer` sends them as email attachments. Files are not stored anywhere after the request finishes. This is fragile for videos and large documents because SMTP providers often reject large messages and failed sends leave no stored copy.

## Proposed Behavior

Keep the existing submission endpoint:

```text
POST /api/send-email
```

For multipart submissions with files, the backend will:

1. Validate the submission as it does today.
2. Create a `submissionId`.
3. Save every uploaded file to local disk.
4. Persist the submission and file metadata to SQLite.
5. Send the email without attachments.
6. Include file links in the email body.

File links will use `API_URL` from the active environment:

```text
stage: https://api-mr-doc.univpro.ru/files/<token>
prod:  https://api.mr-doc.ru/files/<token>
```

There is no duplicated `/api` segment in file links.

## Storage Layout

Use a local storage directory that is not served as a static folder:

```text
storage/
  data.sqlite
  files/
    2026/
      04/
        <submissionId>/
          <fileId>.mp4
          <fileId>.pdf
```

The stored filename is generated by the backend. The original filename is kept only as metadata.

## Database

Use SQLite as an embedded database. This avoids running a separate database service while still making token lookup, future admin views, and backups straightforward.

Initial tables:

```text
submissions
- id
- name
- phone
- reviewText
- service
- privacyConsent
- ip
- createdAt

submission_files
- id
- submissionId
- token
- originalName
- storedName
- mimeType
- sizeBytes
- relativePath
- createdAt
```

`submission_files.token` must be unique.

## File Download Endpoint

Add a separate router for file downloads:

```text
GET /files/:token
```

Behavior:

- Look up `token` in SQLite.
- Return `404` if the token is unknown.
- Resolve the file path under the configured storage directory.
- Reject path traversal by resolving and checking the final absolute path.
- Return the file with `Content-Type`, `Content-Length`, and `Content-Disposition`.

The storage directory must not be exposed through `express.static` or direct nginx static serving.

## Link Security

The site has no authorization, so file access is protected by unguessable links.

Each file receives a long random token, for example 32 random bytes encoded as hex. Direct file paths and sequential IDs are never exposed in emails.

This is not equivalent to authenticated access. It is a pragmatic fit for the current website because clinic staff receive the links by email and the public site has no user accounts.

## Email Format

If files were uploaded, add a files section to both text and HTML emails.

Example text:

```text
Файлы:
- video-review.mp4: https://api.mr-doc.ru/files/<token>
- document.pdf: https://api.mr-doc.ru/files/<token>
```

If there are no files, the email stays functionally the same as today.

## Limits and Validation

Keep the existing `MAX_FILE_SIZE_MB` and `MAX_FILES` settings initially.

The implementation should be ready to add stricter validation:

- allowed MIME types for documents, images, and videos;
- larger separate limit for videos if needed;
- executable file rejection;
- clear client-facing upload errors.

## Backup Requirement

Because files are permanent data, production deployment must back up:

```text
storage/data.sqlite
storage/files/
```

The app should not auto-delete stored uploads.

## Implementation Boundaries

Suggested modules:

```text
src/routes/sendEmail.js
src/routes/files.js
src/storage/database.js
src/storage/fileStorage.js
src/email/sendMail.js
src/utils/formatSubmission.js
```

`sendEmail` should coordinate submission handling, but disk and database logic should live in storage modules so the route does not become a large mixed-responsibility file.
